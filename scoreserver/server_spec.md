# Puzzle Histogram Backend Spec (Go)

## Overview

This document specifies a small Go backend service for a puzzle-based video game. The service accepts player solutions as canonical `80 x 64` PNG files, computes three cost metrics from the submitted solution, stores canonical per-player statistics, and serves histogram data for each puzzle.

The three metrics are:

- `gates`
- `total_ink`
- `core_area`

The backend is designed for community statistics, not adversarial leaderboard-grade security. The goals are:

- dedupe repeated submissions from the same install
- compute metrics server-side from the canonical PNG
- maintain per-puzzle histograms
- allow future rescoring and aggregation changes
- keep deployment simple

Recommended stack:

- Go
- SQLite for v1
- JSON over HTTPS
- server-side configurable bucketing for `total_ink` and `core_area`

---

## Design Principles

1. The submitted PNG is the canonical solution artifact.
2. The server does not trust client-reported metric values.
3. Histograms are based on **best per metric per install**.
4. The server stores append-only submission history plus current per-install state.
5. Histogram aggregation is maintained server-side and returned in compact form.
6. Bucket definitions for `total_ink` and `core_area` are configurable and can be changed later, followed by re-aggregation.

---

## Functional Requirements

The backend must:

1. Accept solution submissions for a specific puzzle.
2. Validate that the uploaded image is a valid PNG with exact dimensions `80 x 64`.
3. Decode the PNG and normalize it for canonical storage and hashing.
4. Compute `gates`, `total_ink`, and `core_area` from the decoded image.
5. Record all submissions in append-only history.
6. Maintain canonical per-install per-puzzle state, including:
   - first submission
   - latest submission
   - best solution for `gates`
   - best solution for `total_ink`
   - best solution for `core_area`
7. Maintain aggregate histogram counts per puzzle and metric.
8. Return histogram data, percentiles, and current canonical values to the client.
9. Support re-aggregation when bucket definitions change.
10. Support rescoring if puzzle logic or scoring logic changes.
11. Include versioning to avoid mixing incompatible submissions.
12. Support light abuse controls such as rate limiting and payload validation.

---

## Non-Goals

The backend does not need to provide, for v1:

- account-based identity
- anti-cheat guarantees against determined attackers
- social/community browsing UI
- real-time streaming updates
- multiplayer/session features

---

## Core Data Model

There are four main storage concerns:

1. Puzzle/version metadata
2. Append-only submission history
3. Current per-player per-puzzle state
4. Histogram aggregates

### Identity Model

Player identity is based on a random install ID generated by the client and stored locally.

This install ID is treated as a soft dedupe key, not a secure identity.

Canonical uniqueness for current state is:

- `(puzzle_id, puzzle_version, install_id)`

---

## Metric Canonicalization

The server must not hash raw uploaded PNG bytes directly, because two different encoders can produce different PNG files for the same pixels.

Canonicalization procedure:

1. Decode uploaded PNG.
2. Verify width = 80 and height = 64.
3. Convert to the canonical internal pixel representation required by the scoring code.
4. Compute a content hash over canonical pixel data.
5. Optionally re-encode to a server-canonical PNG for storage.

Definitions:

- `solution_hash`: hash of canonical pixel data
- `canonical_png`: normalized PNG bytes produced by the server

This allows duplicate detection across different uploaded byte streams.

## SQL Schema

The following schema is suitable for SQLite v1.

## Table: `puzzles`

Stores puzzle metadata and versioning.

```sql
CREATE TABLE puzzles (
    puzzle_id            INTEGER NOT NULL,
    puzzle_version       TEXT NOT NULL,
    scoring_version      TEXT NOT NULL,
    title                TEXT,
    is_active            INTEGER NOT NULL DEFAULT 1,
    created_at           TEXT NOT NULL,
    PRIMARY KEY (puzzle_id, puzzle_version)
);
```

Notes:

- `puzzle_version` changes when the puzzle definition changes.
- `scoring_version` changes when metric computation changes.
- Histograms should never mix incompatible versions.

## Table: `submissions`

Append-only history of every accepted submission.

```sql
CREATE TABLE submissions (
    submission_id        INTEGER PRIMARY KEY AUTOINCREMENT,
    puzzle_id            INTEGER NOT NULL,
    puzzle_version       TEXT NOT NULL,
    scoring_version      TEXT NOT NULL,
    install_id           TEXT NOT NULL,
    solution_hash        TEXT NOT NULL,
    canonical_png        BLOB NOT NULL,
    gates                INTEGER NOT NULL,
    total_ink            INTEGER NOT NULL,
    core_area            INTEGER NOT NULL,
    submitted_at         TEXT NOT NULL,
    client_game_version  TEXT,
    client_platform      TEXT,
    client_ip_hash       TEXT,
    accepted             INTEGER NOT NULL DEFAULT 1,
    reject_reason        TEXT,
    FOREIGN KEY (puzzle_id, puzzle_version)
        REFERENCES puzzles (puzzle_id, puzzle_version)
);
```

Recommended indexes:

```sql
CREATE INDEX idx_submissions_puzzle_version
    ON submissions (puzzle_id, puzzle_version);

CREATE INDEX idx_submissions_install_puzzle_version
    ON submissions (install_id, puzzle_id, puzzle_version);

CREATE INDEX idx_submissions_solution_hash
    ON submissions (solution_hash);
```

## Table: `player_puzzle_state`

Stores current canonical state for one install and one puzzle version.

```sql
CREATE TABLE player_puzzle_state (
    puzzle_id                 INTEGER NOT NULL,
    puzzle_version            TEXT NOT NULL,
    scoring_version           TEXT NOT NULL,
    install_id                TEXT NOT NULL,

    latest_submission_id      INTEGER NOT NULL,
    latest_solution_hash      TEXT NOT NULL,
    latest_gates              INTEGER NOT NULL,
    latest_total_ink          INTEGER NOT NULL,
    latest_core_area          INTEGER NOT NULL,
    latest_submitted_at       TEXT NOT NULL,

    best_gates_submission_id  INTEGER NOT NULL,
    best_gates_solution_hash  TEXT NOT NULL,
    best_gates                INTEGER NOT NULL,
    best_gates_at             TEXT NOT NULL,

    best_ink_submission_id    INTEGER NOT NULL,
    best_ink_solution_hash    TEXT NOT NULL,
    best_total_ink            INTEGER NOT NULL,
    best_ink_at               TEXT NOT NULL,

    best_area_submission_id   INTEGER NOT NULL,
    best_area_solution_hash   TEXT NOT NULL,
    best_core_area            INTEGER NOT NULL,
    best_area_at              TEXT NOT NULL,

    submission_count          INTEGER NOT NULL DEFAULT 1,

    PRIMARY KEY (puzzle_id, puzzle_version, install_id)
);
```

Notes:

- This table defines the canonical values used in histograms.
- Histogram aggregation is based on `best_gates`, `best_total_ink`, and `best_core_area`.

## Table: `histogram_counts`

Stores aggregate counts per puzzle and metric.

```sql
CREATE TABLE histogram_counts (
    puzzle_id             INTEGER NOT NULL,
    puzzle_version        TEXT NOT NULL,
    scoring_version       TEXT NOT NULL,
    bucket_set_version    TEXT NOT NULL,
    metric_name           TEXT NOT NULL,
    min_value             INTEGER NOT NULL,
    max_value             INTEGER,
    count                 INTEGER NOT NULL,
    updated_at            TEXT NOT NULL,
    PRIMARY KEY (
        puzzle_id,
        puzzle_version,
        scoring_version,
        bucket_set_version,
        metric_name,
        min_value
    )
);
```

Notes:

- Each row represents one bucket for one metric. `min_value` is inclusive; `max_value` is exclusive and may be NULL for an open-ended top bucket.
- `bucket_set_version` identifies the bucket configuration that produced these rows. Changing bucket definitions requires bumping this version and rebuilding.
- Bucket range values come from the server config file.

## Histogram Semantics

### Canonical Values Used

Each histogram uses one value per install:

- `gates`: `best_gates`
- `total_ink`: `best_total_ink`
- `core_area`: `best_core_area`

This means each metric is aggregated independently.

## HTTP API

All endpoints use JSON.

Base URL example:

- `/api/v1`

## Endpoint: `POST /api/v1/submit-solution`

Submit a solution PNG for scoring and aggregation.

### Request

```json
{
  "puzzle_id": 12,
  "puzzle_version": "b83f2b1c",
  "install_id": "f84f5f59-9aef-47b2-a9a1-86fa1d894d4d",
  "client_game_version": "1.0.3",
  "client_platform": "windows",
  "solution_png_base64": "..."
}
```

### Validation

The server must:

1. Check required fields are present.
2. Verify `puzzle_id` and `puzzle_version` exist and are active.
3. Decode the base64 payload.
4. Parse PNG.
5. Verify exact image dimensions `80 x 64`.
6. Normalize image.
7. Compute `solution_hash`.
8. Compute `gates`, `total_ink`, `core_area`.
9. Enforce payload limits and rate limits.

### Processing

The server must:

1. Insert a row into `submissions`.
2. Create or update `player_puzzle_state`.
3. Update histogram aggregates if any best metric changed.
4. Return current histogram data and the player's canonical values.

## Optional Admin Endpoint: `POST /api/v1/admin/rebuild-histograms`

Rebuild aggregate histogram counts from `player_puzzle_state`.

This should be protected and not exposed publicly.

### Request

```json
{
  "puzzle_id": 12,
  "puzzle_version": "b83f2b1c",
  "bucket_set_version": "v2"
}
```

### Behavior

- clears existing `histogram_counts` for the specified puzzle/version and bucket_set_version
- scans `player_puzzle_state`
- re-buckets canonical best metric values
- writes fresh aggregates

## Optional Admin Endpoint: `POST /api/v1/admin/rescore-puzzle`

Rescore stored submissions if scoring logic changes.

This should be protected and not exposed publicly.

### Behavior

- scans `submissions` for a given puzzle/version
- recomputes metrics from stored canonical PNG
- updates `player_puzzle_state`
- rebuilds histogram counts

## Required Server Behavior

## Submission Handling Algorithm

For `POST /submit-solution`:

1. Validate request.
2. Decode and validate PNG.
3. Normalize pixels and compute `solution_hash`.
4. Compute `gates`, `total_ink`, `core_area`.
5. Begin transaction.
6. Insert new row into `submissions`.
7. Read `player_puzzle_state` for `(puzzle_id, puzzle_version, install_id)`.
8. If no state exists:
   - create state using this submission as first, latest, and best for all metrics
   - increment histogram counts for all three metrics
9. If state exists:
   - update `latest_*`
   - compare new metric values to current best values
   - for each metric improved:
     - decrement old histogram bucket count
     - increment new histogram bucket count
     - update corresponding best fields in `player_puzzle_state`
   - increment `submission_count`
10. Update summary stats as needed.
11. Commit transaction.
12. Compute or fetch percentiles and return response.

All aggregate updates must happen inside the same transaction.

## Tie Handling for Best Values

Use strict improvement only:

- if new value `< old best`, replace best
- if new value `== old best`, keep current best solution reference unless there is a reason to prefer latest

Recommended v1 choice:

- retain the earlier best on ties

This avoids unnecessary aggregate churn.

## Bucket Configuration

Bucket definitions for `total_ink` and `core_area` should live in server config, not client code.

Recommended config fields:

- `bucket_set_version`
- ordered bucket list for each metric
- each bucket contains:
  - `min_inclusive`
  - `max_exclusive` or null for open-ended
  - display label

The bucket lookup function must be deterministic and total; every allowed value must map to exactly one bucket.

For `gates`, exact integer bins may be implemented directly without config, though a unified bucket interface is still preferable.

---

## Versioning Rules

A submission is only aggregated with data sharing the same:

- `puzzle_id`
- `puzzle_version`
- `scoring_version`
- `bucket_set_version` for histogram presentation

Rules:

1. If puzzle content changes, bump `puzzle_version`.
2. If metric computation changes, bump `scoring_version`.
3. If histogram bucket definitions change, bump `bucket_set_version` and rebuild histogram aggregates.
4. Do not mix incompatible versions in the same returned histogram.

---

## Security and Abuse Controls

The backend is not expected to be cheat-proof, but must include basic hygiene.

Required controls:

- HTTPS only
- request body size limits
- PNG decode limits
- exact dimension checks
- rate limiting per IP
- rate limiting per install ID
- reject malformed base64 / malformed PNGs
- reject unknown puzzle/version combinations
- reject obviously impossible metric values after computation if such constraints exist

Recommended controls:

- hash client IP before storing
- ignore exact duplicate repeat submissions from same install if desired
- structured logging for suspicious request patterns
- CORS restricted to expected clients if applicable

---

## Performance Considerations

Expected request cost is low because solution PNGs are under 2 KB.

Recommended strategy:

- decode and score synchronously during submit
- update histogram counts incrementally rather than recomputing from scratch
- keep histogram counts in SQL
- cache hot histogram responses in memory if needed

SQLite is sufficient for v1 because:

- single server instance
- moderate traffic
- short transactions

Migration path:

- if traffic or deployment complexity increases, move to PostgreSQL with minimal schema changes

## Rebuild and Rescore Procedures

## Histogram Rebuild

Used when bucket definitions change.

Procedure:

1. Load target bucket set.
2. Read all `player_puzzle_state` rows for the puzzle/version.
3. For each row:
   - map `best_gates`, `best_total_ink`, `best_core_area` to buckets
   - accumulate counts
4. Replace corresponding `histogram_counts` rows transactionally.

## Full Rescore

Used when scoring rules change.

Procedure:

1. Read all `submissions` rows for the target puzzle/version.
2. For each submission:
   - decode stored canonical PNG if needed
   - recompute metrics
3. Rebuild `player_puzzle_state` from submission history.
4. Rebuild histograms from new canonical state.
5. Update `scoring_version` where appropriate.

## Summary of Key Decisions

- Use Go.
- Store the canonical PNG, not just metric values.
- Compute metrics on the server.
- Use install ID as a soft dedupe key.
- Base histograms on best per metric per install.
- Use exact bins for `gates`.
- Use config-driven buckets for `total_ink` and `core_area`.
- Store append-only submissions plus current per-player state.
- Maintain histogram aggregates incrementally.
- Support later rebuild/rescore operations through versioned metadata.
